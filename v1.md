# Wysa for Teams — V1 Product Requirements

> V0 established the flows, the persona, and the calendar-awareness concept. V1 makes it real — real calendar data, real users, real context inference.

---

## What changes in V1

V0 used mock calendar data and simulated a Microsoft Teams environment. V1 connects to real calendar APIs, runs inside an actual Teams or Slack integration, and uses meeting context to make responses feel genuinely personal rather than generically smart.

The core flows (UC1 morning check-in, UC2 overwhelmed) stay the same. What changes is the intelligence layer underneath them.

---

## Calendar integration

### What to connect
- Google Calendar API (OAuth 2.0)
- Microsoft Graph API (for Teams-native deployment)
- Scope: read-only. Never write without explicit user action.

### What to pull
- Today's events: title, start time, end time, attendees
- Next 3 days: titles and times only (for triage and deadline planning)
- OOO / focus time blocks already set by user
- Recurring meetings (to detect pattern load, not just today's snapshot)

### What to compute server-side before every session
```
freeSlots[]         — gaps between events, filtered to ≥20 min
backToBacks[]       — consecutive meetings with <10 min gap
longestBlock        — single largest free slot
meetingLoad         — total meeting time as % of working hours
dayType             — classified: focused / fragmented / meeting-heavy / open
tomorrowSlots[]     — next available deep work windows
```

All of this gets injected into the LLM context at session start. The LLM never sees raw calendar data — only the computed summary.

## Context inference layer

### Meeting title matching
When a user mentions a task (in triage, assign, or UC2 free text), run a fuzzy match against today's and tomorrow's meeting titles before sending to GPT.

If match confidence > 70%: inject into system prompt as a soft signal.
```
"User mentioned 'finish PRD'. Today's calendar has 'PRD Walkthrough' at 3pm — 
this task likely feeds into that meeting. Ask if that's the case before 
assuming. Don't state it as fact."
```

If no match: proceed as normal.

### What this enables
- "Is this for your PRD Walkthrough at 3pm?" instead of generic slot assignment
- "You have a 1:1 with Subhash at 4:30 — want to use 12:30 to prep?" instead of just listing free slots
- Triage that accounts for actual upcoming deadlines, not just today's gaps

### Rules for inference
- Always frame as a question, never an assumption
- If the bot is wrong, the user corrects it — that's fine and builds trust
- Never reference attendee names unless the user mentioned them first
- Inference is a suggestion layer, not a scheduling engine

## Trigger moments

V0 used a manual landing page to pick UC1 or UC2. V1 triggers automatically.

| Trigger | Condition | Flow |
|---|---|---|
| Morning check-in | First open of Teams/Slack between 9–10am | UC1 |
| Pre-meeting gap | 10–15 min before a meeting starts | Micro UC1 — one slot, one task |
| Post-meeting re-entry | Meeting ends, next meeting >20 min away | UC2 lite — "what's loudest right now?" |
| Overload signal | 3+ consecutive meetings detected, gap appears | UC2 full |
| End of day | 30 min before last calendar event | EOD offload (new — see below) |

Triggers fire as a notification in Teams/Slack DM. User can dismiss. No mandatory flows.

---

## EOD offload (new UC)

The moment before close-of-day is the highest-stakes moment the product doesn't yet own.

**Trigger:** 30 min before the last calendar event of the day, or 6pm if calendar is empty after 5pm.

**Flow:**
1. Bot surfaces what's still open from the morning check-in (tasks assigned to slots — were they done?)
2. Asks user to dump anything still in their head — one message, no structure needed
3. Captures it, confirms it's saved, tells them it'll be waiting tomorrow
4. Explicit permission to stop: "That's it for today."

**The hook:** "If you don't dump it here, you're carrying it home. Takes 30 seconds."

This is the strongest habit-formation moment in the product. The morning check-in has competition (email, Slack, coffee). The EOD moment has none — the user is already trying to wrap up and the bot is offering to help them do exactly that.

## Persona refinements for V1

V0 established the persona as "sharp colleague who knows your calendar." V1 deepens it.

**What changes:**
- Bot references specific meeting titles, not just slot times
- Responses vary based on day type — a fragmented day gets different framing than an open day
- Recurring patterns acknowledged over time: "Your Tuesdays are consistently your heaviest meeting day — want to protect Wednesday mornings as a standing rule?"
- Closing messages reference what was actually accomplished, not a generic sign-off

**What stays the same:**
- Never wellness language
- Never clinical framing
- Always performance infrastructure, not therapy
- Max 2 sentences per message
- Buttons only when the conversation demands them

## Privacy architecture

- Calendar data processed server-side, never stored beyond the session
- No individual data visible to managers or HR
- User controls what the bot can see — opt-in to calendar read on first use
- Explicit "forget this session" option at any time
- No training on user data without explicit consent

---

## What V1 does not include

- Manager view (V2)
- Team-level aggregates (V2)
- Slack integration (V2 — Teams first)
- Mobile native (V2)
- Custom tool recommendations beyond current Wysa widget set
- AI-generated calendar events (philosophical choice — user always owns their calendar)
- Any form of mood tracking or sentiment scoring

## Open questions for V1 scoping

1. **Calendar write access** — deep links work for prototype but in-app event creation is more powerful. What's the trust threshold for users to grant write access? Does framing it as "protect this slot" rather than "create an event" change the answer?

2. **Trigger fatigue** — five trigger moments is a lot. Which two does V1 launch with? Morning check-in is non-negotiable. EOD offload has the strongest habit formation case. Post-meeting re-entry is the most novel. Pick two.

3. **Real names in context** — using attendee names from calendar titles (e.g. "your 1:1 with Subhash") makes responses feel personal but raises privacy questions. Is this opt-in or default-on?

4. **Session memory** — does the bot remember what was said yesterday? Across the week? A bot that says "you said Tuesday was your hardest day last week — still true?" is a different product from one that starts fresh every morning. Decide before building.

5. **Wysa tool set** — current tools in the widget are generic. For V1, are there tools that could be contextualised to the work setting more explicitly? Or is the current set sufficient with better prompting?

## Success metrics for V1

- **Daily active use rate** — target: 40% of users open at least one flow per day by week 4
- **Flow completion rate** — target: 70% of opened flows reach a closing message
- **Tool open rate** — target: 25% of users who are offered a tool open it
- **EOD offload adoption** — target: 30% of users use EOD flow at least 3x per week by week 4
- **Qualitative: "it knows me" moments** — tracked via post-session 1-tap rating (did this feel personal? yes/no)
